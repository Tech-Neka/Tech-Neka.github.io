VIEWS, CONSTRAINTS, INDEXES

VIEWS

Views are temporary tables created from the basic tables to highlight specific designations or features or to show a broad range of features as required by the company or the analyst. They make analysis easier. Creating a view requires a lot more specification or precision than creating a simply table and a lot more rules apply. For example the table headers or column titles must have unique names, as must the View itself. Furthermore, some clauses like the 'ORDER BY' clause cannot be used in the query unless the 'TOP' aggregate is used in the SELECT statement. A View can be a union of other Views or Tables as in the Roxy example below.


CONSTRAINTS

Constraints are checks set up in tables to preserve the integrity of the data, that is, to ensure that only data that meet the set specifications are allowed into the specified column. They can be created with the table initially, or added to an already existing table. The syntax used in both cases are slightly different.

CHECK constraints were put in the fields/columns of the FOREX Sales tables representing BTA and PTA which have a limit of $5000 and $4000 respectively. They were equally put in the 'CBN Purchases' 'Buy_Amount' column as this Amount has a cap of $50000, as shown in the Roxy figure below.

UNIQUE constraints were also put in various fields including item ID's that are not Primary Keys


INDEXES

Indexes are a speedup mechanism to quickly find important or significant features of data within database tables or columns. They are automatically placed in columns that are Primary Keys to spontaneously identify them as they are the links to other tables when JOINS are required.
In this project, Indexes were placed in relevant tables and columns for quick reference such as the Customer table and the Forex Purchases tables to prioritise significant information. A Roxy example illustrates this below.


STORED PROCEDURES

Stored Procedures are mini software programs written in SQL language using statements or queries, that can be stored in the database for re-use. They allow encapsulation of complicated or frequently used logic, and improve query performance by utilizing cached query plans.They are thus created to perform daily, mundane or necessary tasks required very often by the business or analyst and can be and executed or called up whenever desired. For example, a Bureau de Change manager or CEO would require daily updates of the business' performance and to this end would want to know the daily rates and sales figures including the total and average sales and also the profit earned each business day.
